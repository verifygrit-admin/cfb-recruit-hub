MapView.jsx → api.js → Apps Script Web App
  Connection method: fetch (GET), URL = VITE_API_BASE + ?action=db
  Status: FUNCTIONAL
  fetchSchools() in api.js calls ?action=db. App.jsx fetches on mount.
  The school data is loaded once and cached in React state for the session.
  fetchTracker() calls ?action=tracker; result is merged into runQuickList via
  the trackerMap parameter.
  Note: VITE_API_BASE is injected at build time from secrets.VITE_API_BASE in
  deploy.yml. If the secret is missing or stale the entire app shows a data
  error banner. No runtime fallback.

App.jsx → saveRecruit/updateRecruit → Code.gs → Recruits tab
  Connection method: POST (via api.js post() helper)
  Status: FUNCTIONAL
  The post() helper appends all scalar fields to the URL query string so that
  GET-converted redirects still carry the action parameter. Complex fields
  (awards object) are serialized in the JSON body only.
  Note: The apps-script redirect conversion from POST to GET means the JSON body
  is dropped on redirect and only the URL params are parsed in doGet. saveRecruit
  and updateRecruit are only routed in doPost, so they do not have fallback doGet
  handlers — if a redirect strips the body they will receive an "Unknown action"
  error.

App.jsx → validateToken → Code.gs Auth tab
  Connection method: fetch (GET), URL params = said + token
  Status: FUNCTIONAL
  Called on mount if localStorage has cfb_said and cfb_session_token.
  Rolling 180-day expiry refreshed on every validateToken call.
  Session is fully invalidated on signOut.

App.jsx → saveShortList/getShortList → Code.gs → SL-{SAID} tab
  Connection method: POST (saveShortList), GET (getShortList)
  Status: FUNCTIONAL
  saveShortList is debounced 2 seconds in App.jsx (line 228). Only fires when
  auth is present. getShortList is called on auth completion and on mount
  if localStorage is empty for the SAID.

App.jsx URL param handling → Code.gs:completePendingAccount /

  Connection method: URL query params (?completeSAID=&token= and ?changeEmail=&token=)
  Status: FUNCTIONAL (code paths present and complete)
  Both URL param handlers strip params from the URL via replaceState immediately
  on mount. The completeSAID path triggers AuthModal in createAccount view with

MapView.jsx — Interactive map component
  Renders all 661 schools on a dark CARTO tile background using Leaflet +
  leaflet.markercluster. In browse mode: markers colored by athletic tier
  (DIV_COLORS from constants), filtered by sidebar state. In quicklist mode:
  only matched schools shown (top/good/borderline color coding), non-matching
  schools hidden entirely. Cluster groups use dark forest green styling.
  Each marker has a popup with school name, location, conference, selectivity,
  ADLTV, ADLTV rank, admission rate, graduation rate, COA, merit aid, and
  (in quicklist mode) match rank, academic fit, distance, and optional net
  cost/DROI. Q and Staff links appear if populated. A "Add to Short List"
  button calls window.__toggleShortList(unitid).
  ⚠️ Short List button in popup uses global window.__toggleShortList — this
  is a bridge pattern required because Leaflet popup HTML cannot call React
  state directly. The global is updated in a useEffect in App.jsx. If the
  effect runs after a popup is open but before the user clicks, the closure
  will be stale. The effect dependency array (line 272) includes shortList,
  results, schools, auth, said, so this should be fresh on any relevant state
  change, but there is no debounce guard.
  State management: re-renders entire cluster layer (clears and recreates)
  on every change to schools, results, mode, filters, or shortListIds.
  Legend overlaid bottom-left in quicklist mode; no legend in browse mode
  (legend is in Sidebar).

Sidebar.jsx — Browse-mode filter panel
  Left drawer on mobile, persistent aside on desktop. Filters: Division Tier
  (toggle buttons), Admissions Selectivity (toggle buttons), Conference
  (dropdown), ADLTV Rank Top-N (buttons), State (multi-select chips), free

  Visible count shows "Showing X of Y programs" derived from App.jsx useMemo.
  The visible count in Sidebar reflects the memo'd calculation which exactly

Mobile responsiveness

  media queries (not read in full but referenced). Sidebar uses a drawer pattern
  on mobile (drawerOpen state + drawerOverlay div). ModeToggle is a compact
  dropdown. MapView disables Leaflet's tap emulation (tap: false) to prevent
  double-fire on mobile.

Short list persistence UI
  Short List panel gates on said (not auth) — a user with a SAID but no auth
  can see the gate "Create Your GRIT Fit Profile First" message.
  ⚠️ The ShortList component receives said as a prop. If a user has a SAID
  (from a prior submit) but no auth, said is truthy and ShortList will NOT
  show the "Create Profile" gate — it will show either the list or the empty
  state. Adding schools to the list from this state is blocked at the
  window.__toggleShortList level (redirects to GRIT Fit form), but the
  ShortList panel itself shows as an accessible mode.

⚠️ CRITICAL: Recruits tab cols 24–26 naming collision
  saveRecruit (Code.gs line 133) defines the Recruits tab header row with:
    col 24 = "status", col 25 = "pendingToken", col 26 = "pendingTokenExpiry"
  updateRecruitAuthFields (Code.gs lines 365–367) conditionally writes headers:
    col 24 = "lastLogin", col 25 = "lastLogout", col 26 = "loginCount"
  These two functions disagree on what cols 24–26 contain. When a new recruit
  profile is created (saveRecruit), cols 24–26 hold status/pendingToken/expiry.
  When an auth event fires (login/logout), updateRecruitAuthFields checks if
  headers are blank at those positions — but they are NOT blank (they have
  status/pendingToken/pendingTokenExpiry written by saveRecruit). So the
  conditional header write on lines 365–367 never executes, and the actual
  write on lines 370–375 overwrites status/pendingToken/pendingTokenExpiry
  with timestamps and login counts.
  Result: After any login/logout event for a recruit, col 24 becomes a
  timestamp string (lastLogin), col 25 becomes a logout timestamp, and col 26
  becomes a count integer — replacing what should be the pending/active status.
  This means completePendingAccount (which reads cols 24–26 for the token)
  may find a timestamp where it expects a UUID token.
  markRecruitActive (Code.gs lines 562–575) also writes to col 24–26 directly
  (status="active", clears 25–26). This function is called from createAccount,
  which fires after completePendingAccount, so the active status should survive —
  but the loginCount overwrite from updateRecruitAuthFields called immediately
  after (line 419: updateRecruitAuthFields(said, "login")) then writes
  timestamps to cols 24–26 again, destroying the "active" status just written.
  This is a structural data integrity bug that will corrupt the Recruits tab.

⚠️ Gate-check.mjs uses "G5" tier key, production uses "G6"
  gate-check.mjs defines ATH_STANDARDS with "G5" as the second tier and
  TIER_ORDER as ["Power 4","G5","1-FCS","2-Div II","3-Div III"].
  constants.js uses "G6" for the same tier in all three places.
  The school.Type field in the GrittyOS DB uses "G6" (evidenced by the
  consistent references in scoring.js and MapView.jsx).
  As a result, gate-check.mjs would find 0 schools matching Gate 1 for the
  G5 tier because school.Type === "G6", not "G5". Any result from gate-check.mjs
  should be treated as unreliable. pos-check.mjs has the same issue.

⚠️ SettingsPage email change flow is misleading
  SettingsPage.jsx calls requestEmailChange (api.js line 154), which maps to
  Code.gs:requestEmailChange. This function changes the email immediately in
  both the Auth tab and the Recruits tab — no confirmation code, no magic link.
  The UI button says "Send Confirmation Link →" and the success state says
  "Email updated!" and instructs the user to "Sign out and back in to see the
  update." A user could reasonably believe a confirmation step occurred when
  it did not. The email changes the moment they click the button, even if the
  new address is a typo. This could lock a user out of their account.
  Additionally: api.js also exports requestEmailChangeMagicLink (lines 167–172),
  a confirmed email change flow with a magic link. SettingsPage does not use
  this function. It is unclear which path was intended for production.
  Evidence: SettingsPage.jsx lines 69–82; Code.gs lines 642–668.

⚠️ saveRecruit called without awaiting on GPA-fail and no-top50 paths

    saveRecruit({ ...parsed, timestamp: ... }).catch(() => {});

⚠️ handleSubmit: GPA-fail path with unauthenticated user calls saveRecruit

⚠️ Short List toggle requires auth but short list panel is accessible without auth
  window.__toggleShortList (App.jsx) gates on auth and redirects unauthenticated
  users to GRIT Fit form or sign-in modal. However, ShortList.jsx gates only on

  they can view items. On logout, setShortList([]) is called, so this is mostly

⚠️ Alert() used for short list limit notification
  App.jsx line 257: alert("Your Short List is full (max 40 schools)...")
  Blocking native browser alert is not appropriate for a polished UI and breaks

⚠️ SwitchModal is rendered inline in App.jsx with no dedicated component
  The switch-account modal (rendered when showSwitchModal is truthy) and the

⚠️ handleSubmit: isNewPerson logic requires newProfileMode AND email mismatch

  of the form email (line 555: email: auth.email). No warning is shown. This is

⚠️ MapView re-creates entire cluster layer on every shortListIds change
  MapView.jsx line 50: clusterRef.current is removed and recreated on every
  dependency change. shortListIds is a useMemo array that changes identity
  whenever shortList changes. Adding/removing one school from the short list
  causes a full re-render of all 661 (or filtered N) markers. At scale this
  is measurable performance degradation. The current maxClusterRadius of 40
  and the school count of 661 make this tolerable but not ideal.
  A more efficient approach would be to update only the affected popup's button
  state without re-creating the cluster.

⚠️ TESTING_FLOWS.txt Flow 3 describes a 6-digit email verify modal that does
  not exist in the frontend
  The TESTING_FLOWS.txt description says:
    "verify modal appears ('check your email for a 6-digit code')"

  The Auth cols 13–15 are labeled "pendingNewEmail/emailVerifyCode/emailVerifyExpiry"

  (requestEmailChangeMagicLink / confirmEmailChangeMagicLink). The two paths

⚠️ resendSetupEmail backend reads cols 1–24 but pendingToken is col 25
  Code.gs:resendSetupEmail (line 620): getRange(2, 1, lastRow-1, 24).getValues()
  reads 24 columns. The function then checks r[23] for status ("pending").
  This is correct — col 24 (0-indexed col 23) is status. However, after
  finding the row, it writes the new token to sheet.getRange(i+2, 25) and
  expiry to getRange(i+2, 26), which are cols 25 and 26 — those are
  pendingToken and pendingTokenExpiry respectively. The read uses 24 columns
  but the writes go to 25 and 26, which are just outside the read range.
  This is technically correct since they are separate write calls, not reads.
  However, the read range could be widened to 26 for consistency.

⚠️ validateToken in App.jsx mount effect does not handle network failure
  The validateToken call in the mount useEffect (App.jsx lines 88–126) uses
  .catch(() => {}) — silently ignores network errors. If the network is
  unavailable on load, the user will see no results and no session, even if
  they have a valid cached SAID and would be served from localStorage. The
  session recovery silently fails with no user-visible feedback.

⚠️ restoreSessionRef pattern — deferred scoring is silently dropped on edge cases
  The restoreSessionRef.current approach (App.jsx) stores a profile for later
  scoring when schools haven't loaded yet. If fetchSchools fails (setDataError),
  the schools array remains empty and the restore useEffect (line 158–181) guard
  condition (!schools.length) prevents scoring from ever running. The user
  signs in, sees no results, and gets no explanation that the data load failed.
  The dataError banner would be shown, but the connection between "data failed
  to load" and "your results can't be restored" is not surfaced to the user.

⚠️ Short list server restore has two code paths for the same logic
  The inline getShortList call in handleAuthComplete (lines ~397–421) and the
  useEffect triggered by restoreSlRef.current (lines 184–219) both map server
  short list data back to school objects. They share the same mapping code but
  are written separately. If a field mapping is changed in one, it must be
  updated in both. The inline version (handleAuthComplete) does not map
  _schoolName, _divLabel, _coaNum, _qLink, _coachLink in the same explicit way
  as the useEffect version — it relies on the subsequent "resolve missing derived
  fields" useEffect (lines 297–316) to fill those in. This is functional but
  creates an indirect dependency.

⚠️ QuickListForm and SettingsPage import a hardcoded team photo
  Both QuickListForm.jsx (line 3) and SettingsPage.jsx (line 4) import
  "../assets/bchigh-team.jpg" as a background image. This is a BC High-specific
  football team photo. This asset should be replaced with a generic background
  or made configurable if this platform is intended to serve multiple schools
  or the general public. It is a brand-specific artifact in a general-purpose
  platform.

⚠️ No automated tests of any kind
  There are no test files (*.test.js, *.spec.js, __tests__/ directory) in the
  project. gate-check.mjs and pos-check.mjs are manual diagnostic scripts run
  against the live API. The testing burden is entirely manual per TESTING_FLOWS.txt.
  With 6+ untested auth flows and a complex state machine in App.jsx, regressions
  are hard to catch.

⚠️ Apps Script has 20,000 requests/day rate limit on free Google Workspace
  ROADMAP.txt documents this. The current deployment routes all DB reads,
  all auth operations, and all short list operations through a single Apps
  Script Web App. Each page load generates at minimum 2 calls (db + tracker).
  There is no caching layer, no CDN for the school DB, and no request
  deduplication. This will hit the limit at moderate traffic.

⚠️ No input sanitization in Code.gs
  Profile fields (name, highSchool, email, etc.) are written directly to
  Google Sheets with no sanitization. While Sheets formula injection via
  "=FORMULA" is not a significant risk in this context (the data is not
  re-executed), unusually long or specially-formatted strings could affect
  sheet formatting or downstream processing.

⚠️ The panel toggle (Map/Table) for quicklist results is only visible when
  results are present (App.jsx line 696). If the user navigates to quicklist
  mode without results, the toggle is hidden and there is no obvious path to
  the form from the results bar. The "Update My Profile" button in the summary
  bar leads to Settings, not back to the QuickListForm. Users who want to
  re-enter their profile from scratch (without using "New Profile" button)
  have no clear entry point once results are showing, other than via the
  Settings page.
