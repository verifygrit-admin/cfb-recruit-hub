BL-008 | 2026-03-23 | Scout (Test 4 setup — schema audit) | MEDIUM | Open
  MORTY SCOPE — separate schema session required.
  profiles.email has no UNIQUE constraint at the database level.
  saveRecruit uses .insert() (not .upsert()), meaning a second
  form submission with the same email address will create a
  duplicate profiles row rather than routing to updateRecruit.
  Scope of impact: any real athlete who submits the form twice
  under the same email currently receives two separate profile
  rows in the profiles table. The isNewPerson routing in App.jsx
  is an app-level guard only — it does not prevent duplicates at
  the database layer if the guard is bypassed or fails silently.
  Fix requires: (1) a new migration adding UNIQUE constraint on
  profiles.email, (2) Chris's approval before the migration runs
  (schema change on a live table), (3) Morty to audit the full
  impact surface — RLS policy interactions, Edge Function behavior
  on conflict, and whether .insert() should be converted to
  .upsert() with a conflict target across any code paths.
  Context: surfaced during Test 4 setup step, 2026-03-23 session.
  Scout ruling: INSERT vs upsert gap is a schema-level risk, not
  an app-level fix. Do NOT implement until Morty audit is complete
  and Chris approves the migration approach.
  File: supabase/migrations/ (new migration, not yet drafted)
  Related: DEC-CFB-007 (RLS policy set), DEC-CFB-008 (Morty PASS gate)
  Do NOT touch until explicitly scheduled.
