── FLOW 1: Submit New Profile While Logged In (Different Email) ──────────────
Goal: Logged-in user submits GRIT Fit form with an email that differs from
      their current session email → backend creates a new SAID via saveRecruit
      → frontend shows AuthModal for new account creation → on success shows
      SwitchModal asking whether to switch to the new account.

NOTE: Updated 2026-03-22 — newProfileMode entry point required.
      The form email field is silently overridden with auth.email on a
      normal logged-in submit (App.jsx:553). A different-email new SAID
      is only created when newProfileMode=true, which requires clicking
      "New Profile" first. Without this step, the form always routes to
      updateRecruit using auth.email.

Steps:
  1. Sign in with your existing account (e.g. email-A)
  2. Run the GRIT Fit form and get to the results view
  3. Click "New Profile" button (top of results / form area) — this
     clears the form and sets newProfileMode=true
  4. Fill out the blank form using a different email (email-B) —
     all required fields; use a real email you can check
  5. Submit
  6. Expect: AuthModal appears prompting account creation for email-B
  7. Create password for email-B account
  8. Expect: SwitchModal — "Switch to email-B?" with Yes/Stay options
  9. Test Stay: confirm session remains as email-A
  10. Re-run steps 1–8, then test Switch: confirm session logs in as email-B

Pass criteria:
  - New SAID created in Recruits tab with status "pending" → "active"
  - New row in Auth tab for email-B
  - SwitchModal appears with correct email shown
  - "Stay" keeps session as email-A; "Switch" logs in as email-B
  - Normal form submit while logged in (without New Profile click) does
    NOT create a new SAID — routes to updateRecruit with auth.email

Status: [x]
Notes: 2026-03-22 — Passed after bug fix (commit 52f0e4e). savedAthleteRef/
       savedResultsRef were captured at submit time (wrong data). Fixed by
       moving capture to handleNewProfile() before blanking. Stay correctly
       restores original user's form, results, and map. Switch path confirmed
       unaffected. Both Stay and Switch tested by Chris Conroy.


── FLOW 2: Magic Link Account Completion ─────────────────────────────────────
Goal: Magic link path activates a PENDING account for a user who did not
      complete the immediate AuthModal after form submission. This is the
      fallback for users who closed the modal or came back later. It is NOT
      the forgot-password flow (that is Flow C / resetPassword). It is NOT
      the same as the immediate createAccount path tested in Flow 1.

      The magic link is one-time use: if createAccount already ran for the
      same SAID, completePendingAccount will fail ("account already exists").
      The AuthModal MUST be dismissed before the link is used.

NOTE: Updated 2026-03-22 — original spec did not require dismissing the
      AuthModal first. Test design corrected after live testing revealed
      that immediate createAccount path exhausts the pendingToken.
      MailApp delivery to Google Workspace confirmed unreliable (silent
      failure — no error thrown). Use Option B (sheet lookup) for testing.

Steps:
  1. Sign out if currently signed in
  2. Submit GRIT Fit form as a logged-OUT user with a fresh email address
  3. When the AuthModal appears — DISMISS IT (click X or Cancel)
     Do NOT create a password. Account must remain in "pending" status.
  4. Open GrittyOS DB spreadsheet → Recruits tab → find the new row
  5. Copy the value in col 25 (pendingToken) and col 7 (email)
  6. Construct the magic link manually (all three params required —
     omitting email leaves the display field blank):
     https://verifygrit-admin.github.io/cfb-recruit-hub/?completeSAID=<SAID>&token=<pendingToken>&email=<url-encoded-email>
     Example encoding: athlete@email.com → athlete%40email.com
  7. Paste the link into the browser and navigate to it
  8. Expect: app opens, password-set modal appears (title: "Complete Your
     Account Access") pre-filled with the SAID
  9. Set a password (8+ characters) and submit
  10. Expect: signed in automatically, profile loaded, redirected to results

Pass criteria:
  - completePendingAccount API call succeeds (not createAccount)
  - Auth tab row created with correct SAID and email
  - Recruits tab status → "active"; pendingToken and pendingTokenExpiry
    cleared (cols 25–26 blank after completion)
  - Session established — user is logged in immediately after
  - Modal title reads "Complete Your Account Access" (not "Create Your Account")
    confirming pendingToken prop was passed to AuthModal
  - Email field in modal is pre-filled and read-only (greyed out, not editable)

Status: [x]
Notes: 2026-03-22 — Passed after spec rewrite and UX fixes. Three issues
       found and resolved: (1) spec missing AuthModal dismiss step —
       createAccount exhausts pendingToken before magic link can be used;
       (2) email param omitted from Option B URL — field blank in modal,
       fixed in spec; (3) pending state UX — fields not visually locked,
       actions buried at bottom, email error shown in red — fixed with
       green status banner at top, greyed fields, suppressed error.
       Core magic link path (completePendingAccount) verified working.
       Email field pre-filled read-only confirmed. MailApp delivery to
       Google Workspace unconfirmed — Option B used for testing.


── FLOW 3: Edit My Profile — Email Change ────────────────────────────────────
Goal: Signed-in user edits their profile and changes their email address →
      magic link sent to the NEW email → user clicks link → email updated
      in both Auth tab and Recruits tab.

NOTE: Updated 2026-03-22 — 6-digit code path was removed in v1.1.3.
      Email change now uses magic link only (requestEmailChangeMagicLink /
      confirmEmailChangeMagicLink). Original spec described 6-digit flow
      which no longer exists.

Steps:
  1. Sign in with existing account (email-A)
  2. Open Edit My Profile
  3. Change email field to a new address (email-B) — keep all other fields
  4. Save / Submit
  5. Expect: confirmation message that a magic link has been sent to email-B
  6. Check email-B inbox for GrittyOS email change link
  7. Click the magic link
  8. Expect: app opens, email change confirmed, session email updated to email-B
  9. Sign out and sign back in with email-B to confirm

Pass criteria:
  - requestEmailChangeMagicLink API call succeeds
  - Magic link email arrives at email-B within ~30s
  - confirmEmailChangeMagicLink updates Auth col 2 to email-B
  - Recruits tab col 7 (email) updated to email-B
  - Old email-A no longer works to sign in
  - email-B signs in successfully

── FLOW 5: GPA-Fail / No Top-50 While Logged In ──────────────────────────────
Goal: Signed-in user submits GRIT Fit form but fails the GPA threshold or
      has no top-50 school matches → should call updateRecruit (update existing
      profile) rather than saveRecruit (create new record).

Steps:
  1. Sign in with existing account
  2. Navigate to GRIT Fit form
  3. Enter same email as your account
  4. Set GPA below the minimum threshold (e.g. 1.5) OR ensure no top-50 matches
  5. Submit
  6. Expect: failure/no-results state shown; no new SAID created
  7. Verify Recruits tab — row updated (timestamp refreshed), no duplicate row

Pass criteria:
  - updateRecruit called (check network tab or Recruits sheet — no new row)
  - No AuthModal shown (already authenticated)
  - No new SAID generated
  - Existing row timestamp updated

── FLOW 6: Short List Persistence Across Sign Out / Sign In ──────────────────
Goal: Short list saved to server (SL-{SAID} tab) is restored correctly when
      user signs back in — localStorage sync is gated on confirmed auth.

Steps:
  1. Sign in, add 3+ schools to short list
  2. Confirm short list saves (check SL-{SAID} tab in Google Sheet)
  3. Sign out
  4. Verify short list is cleared from UI
  5. Sign back in with same account
  6. Expect: short list restored from server

Pass criteria:
  - Short list visible after sign-in matches what was saved before sign-out
  - No ghost short list items from previous session in localStorage
  - getShortList API call fires on auth restore

Status: [x] FIXED (commit 25373d9 — sync effect gated on auth)
Notes: Fixed prior to test session. Sync effect was firing before auth confirmed,
       causing localStorage restore to overwrite server data.
